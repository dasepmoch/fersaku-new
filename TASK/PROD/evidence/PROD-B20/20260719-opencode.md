## Summary

Mounted Duitku payment webhook ingress that verifies merchant + callback signature, parses JSON or form-urlencoded bodies, and reuses the existing payment finalization pipeline (`acceptNormalized` → `ProcessEvent` / `applyNormalized`). Provider stored as `DUITKU`, account scope from config (`duitku-primary`), never from body.

## Routes

| Method | Path | Auth |
| ------ | ---- | ---- |
| POST | `/v1/webhooks/duitku` | body signature + merchantCode |
| POST | `/v1/webhooks/duitku/sandbox` | same; mode override SANDBOX |
| POST | `/v1/webhooks/duitku/live` | same; mode override LIVE |

Outside session/CSRF middleware (same pattern as Xendit payment webhooks).

Success response: `200` + `text/plain` body `OK`.  
Reject: `401`/`400`/`413`/`415` + body `FAILED` (rejection table only; no money mutation).

## Callback signature algorithm

```text
signature = MD5( merchantCode + amount + merchantOrderId + apiKey )
```

- Concatenation is plain string (no separators).
- `amount` is the posted amount string (whole IDR, no decimals); field may be `amount` or `paymentAmount`.
- Digest is **lowercase hex** (32 chars).
- Verification: constant-time compare of computed vs provided (case-normalized to lower).
- Merchant: `merchantCode` must match configured `DUITKU_MERCHANT_CODE` via MD5-digest constant-time compare (length-safe).

Helpers:

- `adapters/duitku.CallbackSignature` / `VerifyCallbackSignature` / `MerchantCodeEqual`
- Application layer duplicates verify formula without importing adapters (layering rule)

## Status map (`resultCode`)

| resultCode | Domain |
| ---------- | ------ |
| `00` | PAID |
| `01` | PENDING (or FAILED if message cancel/fail) |
| `02` + cancel message | CANCELLED |
| `02` | FAILED |
| message expired | EXPIRED |

`provider_event_id`: prefer `reference:resultCode`; fingerprint if empty.  
`external_id`: `merchantOrderId` (resolvePayment).  
`provider_reference`: `reference` or merchantOrderId.
