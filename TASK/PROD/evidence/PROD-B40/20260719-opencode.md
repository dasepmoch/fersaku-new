## Summary

Composition already selected Duitku QRIS when `PAYMENT_PROVIDER=duitku` (B10/B30). B40 confirms and documents that path, strengthens tests that payment=duitku never wires Xendit as the QRIS port, keeps Xendit disbursement independent, and updates launch/security docs that still said “no Duitku / Xendit-only payment”.

Xendit payment webhook remains mounted for historical/late events (not rejected with 410 — option B not chosen). It is not the primary create path when payment=duitku.

## Composition (confirmed)

```text
PAYMENT_PROVIDER=duitku  → Duitku Real QRIS only (Xendit CreateQRIS not selected)
PAYMENT_PROVIDER=xendit  → legacy Xendit Real QRIS (local/transition)
PAYMENT_PROVIDER=fake    → xendit.Fake QRIS (local)
DISBURSEMENT_PROVIDER    → independent: fake | xendit
```

`wireMoneyProviders` kind strings: `paymentKind` ∈ {fake, xendit, duitku}, `disbursementKind` ∈ {fake, xendit}.

When payment=duitku and disbursement=xendit: `needXenditReal` is true only for disbursement; `qris` is `*duitku.Real`, `disburse` is `*xendit.Real`.

## Option A

- Xendit QRIS adapter code (`adapters/xendit` CreateQRIS) **kept on disk**.
- Not injected into checkout/gateway when `PAYMENT_PROVIDER=duitku`.
- Delete of Xendit QRIS code deferred until canary stable (ADR-0008 §8).
