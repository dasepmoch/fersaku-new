## 1. Rules (hard)

1. **Never** paste secret values into evidence, git, Docker images, or chat.  
2. Production hosts **must not** load `/var/www/pg.txt` (demo-only reference).  
3. Browser never receives `DUITKU_*`, `XENDIT_*`, encryption keys, SMTP passwords, session/CSRF secrets.  
4. Deploy injects from **secret manager** (or platform secrets), not bake-in.  
5. Key-swap at KEY-62 only flips §3 LIVE rows after staging SM already works.

## 4. Populate plan (human ops)

### 4.1 Choose secret manager

Pick one (document choice in this folder when done):

| Option | Notes |
|--------|-------|
| A. Cloud SM (AWS Secrets Manager / GCP Secret Manager / Azure Key Vault) | Preferred for managed deploy |
| B. Platform secrets (Fly/Render/K8s secrets / Doppler / 1Password Connect) | OK if audited + RBAC |
| C. **Not allowed for prod:** `/var/www/pg.txt`, committed `.env`, image ENV bake |

### 4.2 Procedure (staging first)

1. Create secret objects for every **TBD** row in §3 (staging prefix e.g. `fersaku/staging/…`).  
2. Wire deploy/runtime to inject env at process start (no file mount of raw key dumps).  
3. Boot API with `APP_ENV=staging`, `PAYMENT_PROVIDER=duitku`, `DISBURSEMENT_PROVIDER=xendit`, `DUITKU_ENV=sandbox`, `XENDIT_ENV=sandbox`, `MAIL_MODE=smtp`.  
4. Verify: `/health/ready` 200; `POST /v1/webhooks/duitku` empty → 401; no secret in logs.  
5. Clone set for prod prefix `fersaku/production/…` with **sandbox still** until KEY-62.  
6. At KEY-62 only: swap LIVE Duitku/Xendit secrets + `*_ENV=production`.

### 4.3 Prod deploy runbook (secrets only)

```text
1. Image: immutable tag (no secrets in image).
2. Runtime env: from SM injection only.
3. Forbidden: pg.txt, docker -e KEY=value in shell history for prod.
4. Post-deploy: health + webhook mount check (401 empty body).
5. Incident: rotate SESSION/CSRF → revoke-all; rotate provider keys → update dashboard.
```
