### 1. Package `backend/internal/adapters/duitku/`

Implement `ports.QRISProvider`:

| Method | Behavior |
| ------ | -------- |
| CreateQRIS | Call Duitku create/inquiry API for QRIS; map to CreateQRISResult |

**Client design (Real):**

```go
type Real struct {
  MerchantCode string
  APIKey       string // never log
  BaseURL      string // default sandbox https://sandbox.duitku.com
  CallbackURL  string
  ReturnURL    string
  PaymentMethod string // from DUITKU_QRIS_PAYMENT_METHOD, default SP
  AccountScope string // default duitku-primary
  HTTPClient   *http.Client
  Timeout      time.Duration
  Log          ports.Logger
}

func NewReal(...) (*Real, error) // require merchantCode+apiKey
func (r *Real) Name() string
func (r *Real) IsFake() bool { return false }
```

**Duitku API (verify against common v2 merchant API; use sandbox paths):**

Typical patterns (implementer must align with Duitku docs; use these well-known shapes if docs offline):

- Create: `POST {BaseURL}/webapi/api/merchant/v2/inquiry`
  - JSON body fields commonly: `merchantCode`, `paymentAmount`, `merchantOrderId` (= ExternalID), `productDetails`, `email` (optional), `paymentMethod` (QRIS code SP), `callbackUrl`, `returnUrl`, `expiryPeriod` (minutes), `signature`
  - Signature often: `MD5(merchantCode + merchantOrderId + paymentAmount + apiKey)` (lowercase hex) — **document exact algorithm in evidence**
- Status: `POST {BaseURL}/webapi/api/merchant/transactionStatus`
  - Body: merchantCode, merchantOrderId, signature `MD5(merchantCode + merchantOrderId + apiKey)`

Map statuses:

| Duitku | Domain |
| ------ | ------ |
| 00 / Success / paid | PAID |
| 01 / process / pending | PENDING |
| 02 / canceled / failed | FAILED / CANCELLED as appropriate |
| expired | EXPIRED |

CreateQRISResult:
- ProviderReference: Duitku reference / merchantOrderId (prefer stable id for GetPayment)
- QRString / QRImageURL from response fields (`qrString`, `qrUrl`, etc. — map what API returns)
- Status PENDING on create success

**HTTP hygiene:**

- Timeouts (15s default)
- Never log API key or full raw body with secrets
- Classify errors: AUTH_FAILURE (401/signature), TIMEOUT, UNAVAILABLE (5xx), REJECTED (4xx business), INVALID_RESPONSE, UNKNOWN_OUTCOME (timeout after send → RequestSent=true)
- Idempotency: use ExternalID as merchantOrderId

**Fake (optional but good):** if PAYMENT_PROVIDER=fake still uses xendit.Fake, you do **not** need a duitku Fake unless composition wants symmetry. Prefer keep fake on xendit package; Duitku Real only.
