## TASK DETAILS (from 01-P0-DUITKU-CONTRACT.md):

### Key findings to fix:
1. `client.go` uses MD5 for inquiry/status/callback signature — Duitku API now requires HMAC-SHA256
2. Default sandbox mode in adapter — production must fail-closed if env/base URL incoherent
3. `reference` (provider's ID) is being used as `merchantOrderId` in subsequent lookups — these must be separate
4. Callback acknowledgment uses `OK` — verify if `SUCCESS` is required per current API docs
5. Production endpoint should be `https://passport.duitku.com/...`

### Implementation steps:
1. Freeze source-of-truth contract: URL production/sandbox, method, content type, field casing, status code, signature formula (HMAC-SHA256, lowercase hex), callback response, timeout, idempotency. Store URL/doc reference and verification date in test/doc, NOT secrets.
2. Replace Duitku signature helper with HMAC-SHA256 (lowercase hex) for inquiry, callback, and transaction status. Add known-vector test from non-secret fixture and negative tests for MD5/field reorder/wrong amount/wrong merchant.
3. Keep callback parser compatible with legacy format ONLY if provider sends legacy; do NOT accept legacy signature in live without explicit mode/feature boundary. Test both form-urlencoded and JSON body shapes.
4. Change lookup context so original `merchantOrderId` is stored and used for `transactionStatus`; `reference` stored as separate provider reference for audit. Data migration for old records must have mapping/repair report.
5. Make `DUITKU_ENV`, `APP_ENV`, `DUITKU_BASE_URL`, callback URL, return URL coherent at boot. Production must reject sandbox base URL, empty/default sandbox, non-HTTPS, or non-allowlist host. Staging/sandbox must reject passport URL.
6. Verify provider acknowledgement response (`SUCCESS`/documented format) with integration stub that records status and body.
7. Audit Xendit QR code and payout API version, endpoint, body shape, callback token, idempotency header, fallback behavior vs latest Xendit docs. Fallback v1 must not mask v2 non-retryable errors.
8. Add contract tests via `httptest` that assert URL path, method, headers, signature, order ID, amount, response mapping, timeout, 4xx/5xx classification, and no-secret logging.

### Acceptance criteria:
- All signature tests use HMAC-SHA256 vectors matching active documentation; old MD5 vectors no longer valid on live path.
- Create → persist → status lookup proves `merchantOrderId` stays identical to initial ID and `reference` is only provider reference.
- Production config without explicit base URL never selects sandbox; mismatch env/host fails before server listen.
- Callback with valid signature yields HTTP 200 and documented acknowledgement; invalid/replay does not change ledger/order.
- Disposable integration stub proves PAID, PENDING, FAILED/EXPIRED, timeout, duplicate callback, unknown outcome.
- `go test ./internal/adapters/duitku/... ./internal/adapters/xendit/... ./internal/application/...` green with evidence.
