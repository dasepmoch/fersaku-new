## TASK DETAILS (from 06-P1-CAPACITY-MIGRATION.md):

### Key findings to fix:
1. `pool.go:28-37` — default `MaxConns=20`
2. `app.go:269` — same config for all roles, no API/worker env override
3. `topology.md` — calculates API 20, worker 10, 2 replicas each, migrate 4 = 64/80, but actual two workers also 20 = 84/80
4. `migrate.sh` — provides `down`, `drop`, `force`, `goto` without production guard
5. `launch_bootstrap.sh` — defaults to running seed, no production-safe mode besides operator remembering `SKIP_SEED=1`

### Implementation steps:
1. Define pool budget per process role and replica count from single config source. Fail startup if total theoretical + headroom exceeds DB max connections.
2. Add env knobs validated for API/worker/migrate/admin, timeout, statement timeout, pool metrics. Don't rely on default 20 for all roles.
3. Migration guard: destructive commands rejected if `APP_ENV=production` except explicit break-glass token/interactive confirmation not activatable in CI. `up` is forward-only default.
4. Lock migration execution to single job, verify backup/PITR checkpoint, migration version, lock timeout, expand/contract compatibility before rollout.
5. Separate bootstrap local/staging/prod; production must not seed demo personas or use local DATABASE_URL fallback.
6. Test concurrent migrate, failed migration, restart, pool exhaustion, long transaction, rollback image against previous schema.

### Acceptance criteria:
- Capacity worksheet matches runtime config for production replica count with measurable headroom.
- Production migration script cannot `drop/down/force/goto` accidentally; CI can still test commands in disposable DB.
- No local default URL/seed path in production; audit log migration stores version, actor/job, duration, result.
- Integration test proves old app still runs after expand migration and new app does not deploy before head migration.
