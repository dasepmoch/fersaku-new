## TASK DETAILS (from 03-P0-PROXY-RATE-LIMIT.md):

### Key findings to fix:
1. `router.go:108-109,150` — `TrustedProxies` field exists but `app.go:997-1040` doesn't populate it from config
2. `trusted_proxy.go:12-20,37-65` — trusts XFF only if peer in CIDR; without CIDR uses RemoteAddr
3. `limiter.go:21-39,54-75` — Redis limiter uses prefix `rl:ip:` and default 120/min
4. `router.go:145-172` — rate limit applied as global middleware; health, auth, checkout, callback, admin, webhook share same budget

### Implementation steps:
1. Add config `TRUSTED_PROXY_CIDRS` validated strictly at boot production; parsing invalid/empty on LB topology must fail or use explicit direct mode.
2. Populate `RouterDeps.TrustedProxies` from config; expose resolved peer policy in safe diagnostics (no raw header).
3. Unit tests: direct client, trusted LB, spoofed XFF from untrusted peer, multi-hop XFF, IPv4/IPv6, malformed RemoteAddr.
4. Separate rate-limit budget per route class: public read, auth, mutation/checkout, admin, provider callback, health/metrics. Callback protected from abuse but not sharing bucket with user traffic.
5. Key strategy: resolved client IP + route class; if authenticated use subject/store scope per threat model. No raw user-controlled header as key.
6. Policy when Redis unavailable per class; health/readiness must explain degraded state and money mutations must not silently go unlimited.
7. Test two replicas behind real/stub proxy with 20 different client IPs, burst, retry-after, callback storm, Redis failover.

### Acceptance criteria:
- Production startup shows trusted proxy CIDR resolved and test proves real IP used only from trusted peer.
- Two clients behind LB do not share bucket; spoof XFF does not change identity.
- Route callback/auth/checkout has limit and retry-after documented; health not locked by regular traffic.
- Redis error policy tested and alert appears before user-visible outage.
- Log/metric includes route class and hashed/bounded client identity, not PII/raw header.
