## TASK DETAILS (from 05-P0-RELEASE-DEPLOYMENT.md):

### Key findings to fix:
1. Repository only has backend Dockerfile and compose rehearsal — no frontend production deployment artifact, platform manifest, or CD/promotion workflow
2. CI builds image but doesn't prove immutable publication, digest promotion, signature/provenance, SBOM, or automatic migrate/deploy/rollback
3. Runbook `topology.md` and `canary-rollback.md` are procedures, not executable deployment contracts

### Implementation steps:
1. Define release manifest: git SHA, image digest API/worker/frontend, migration head, Node/Go versions, base image digest, feature/domain source map, config schema version.
2. Build-once pipeline: test → SBOM/vuln scan → sign provenance → publish immutable digest → deploy staging → smoke/E2E → manual prod approval → canary → promote.
3. Frontend artifact actually run on target (standalone Node/container or platform contract): health endpoint, cache/static policy, env build-time vs runtime, source map policy.
4. Separate migration job from API/worker boot. Gate rollout on migration success, readiness, synthetic, callback ingress, scanner, storage, mail, queue checks.
5. Canary/rollback command selecting previous digest; schema forward-compatible, no auto-down-migrate.
6. Deployment smoke proving callback routes don't 404 on actually-published image, not just source tree.

### Acceptance criteria:
- One release manifest can be retrieved and maps tested artifacts to deployed artifacts.
- Staging and production promotion doesn't rebuild different source; immutable digest verified.
- Canary one slice has rollback tested for API, worker, frontend; rollback completes without losing outbox/payment state.
- Migration failure stops rollout; readiness fails if required dependency not ready.
- Evidence stores digest, health/synthetic result, migration version, deploy time, rollback drill, reviewer.
